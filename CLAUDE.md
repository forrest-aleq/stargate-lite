## Project Overview

Stargate Lite is the execution layer ("The Hands") for the Aleq MIND ecosystem. It provides a capability-based abstraction layer over multiple third-party business services, handling multi-tenant OAuth credential management and secure token storage.

**Integrated Services** (January 2026):
- QuickBooks Online (accounting, vendor management, journal entries, reports)
- Stripe (payments, payouts, balance, transfers, Connect)
- Bill.com (AP automation, vendor payments, bill approvals)
- NetSuite (ERP, journal entries, vendor bills, SuiteQL)
- Sage Intacct (mid-market ERP, GL, AP, AR, cash management)
- HubSpot (CRM, contacts, deals, companies)
- Gmail (email automation)
- Slack (messaging, channels, notifications)
- Gusto (HR, payroll, employees, benefits)
- Shopify (e-commerce, orders, products, fulfillment)
- Square (payments, orders, customers, inventory)
- DocuSign (e-signatures, envelopes, templates)
- Airtable (spreadsheet-database, bases, records)

**Key Architecture Pattern**: The Brain (Aleq MIND) sends abstract capability keys (e.g., `vendor.create`), and Stargate Lite resolves these to concrete tool implementations (e.g., `quickbooks.create_vendor`) and executes them.

**Total Capabilities**: 570+ endpoints across 22 platforms

## Core Architecture

### Request Flow
1. **Request arrives** at `/api/v1/execute` with a `capability_key`, `org_id`, `user_id`, and `args`
2. **Registry lookup** in `app/registry.py` maps capability to handler function
3. **Credential retrieval** from database (app/database.py) using org_id/user_id/service composite key
4. **Token refresh** if OAuth token is expired (connector handles this)
5. **Tool execution** by connector (app/connectors/*)
6. **Response formatting** with structured outputs, logs, and status

### Multi-Tenant Credential Isolation

Credentials are stored with a composite key: `{org_id}:{user_id}:{service}`

Each connector retrieves credentials via:
```python
cred = CredentialManager.get_credential(org_id, user_id, service)
```

This ensures org/user isolation - credentials are never shared across tenants.

### Encryption

All OAuth tokens are encrypted at rest using Fernet symmetric encryption. The `ENCRYPTION_KEY` environment variable must be set in production. The key is generated by `setup.sh` and stored in `.env`.

### Adding New Capabilities

1. **Create or extend connector** in `app/connectors/`:
   - Implement the tool method (takes org_id, user_id, args)
   - Handle credential retrieval and token refresh
   - Return structured dict with outputs

2. **Register in capability registry** (`app/registry.py`):
   ```python
   "capability.key": {
       "handler": connector.method_name,
       "tool_name": "service.method_name",
       "description": "Human readable description",
       "requires_oauth": True/False,
       "service": "service_name"
   }
   ```

3. The capability is now available via `/api/v1/execute`

### OAuth vs API Key Services

- **OAuth services** (QuickBooks, HubSpot, Gmail, Slack): Require per-user token storage and refresh logic
- **API key services** (Stripe): Use global API key from environment variable

Connectors handle both patterns. Check `requires_oauth` in registry.

## Important Files

- **app/main.py**: FastAPI app, main execution endpoint, API key auth
- **app/registry.py**: Capability → Tool mapping (this is the "brain" of the routing)
- **app/database.py**: Encrypted credential storage using SQLAlchemy + Fernet
- **app/models.py**: Pydantic request/response models
- **app/connectors/*.py**: Individual service connectors with tool implementations
- **app/utilities/*.py**: Cognitive utilities (summarization, search, calculations)

## Credential Management

Credentials are stored programmatically:

```python
from app.database import CredentialManager

CredentialManager.store_credential(
    org_id="org_123",
    user_id="user_456",
    service="quickbooks",
    access_token="eyJ...",
    refresh_token="1//...",
    token_expiry=datetime(2025, 10, 19),
    realm_id="9130123456789"  # QuickBooks-specific
)
```

Token refresh is handled automatically by connectors when tokens are near expiry (within 5 minutes).

## Production Considerations

1. **Database**: Switch from SQLite to PostgreSQL by updating `DATABASE_URL`
2. **Secrets**: Use strong `API_SECRET_KEY` and never commit `ENCRYPTION_KEY`
3. **OAuth callbacks**: Update redirect URIs to production domain with HTTPS
4. **CORS**: Restrict `allow_origins` in `app/main.py` middleware
5. **Credential backup**: Encrypted database contains all OAuth tokens - back up regularly

## Code Review & Auditing

When auditing, reviewing, or fixing code/docs, always do a thorough, exhaustive pass the FIRST time. Cross-reference every claim against actual code. Do not treat issues as "doc-only" without verifying the code matches. Never require the user to ask for more rigor — assume "ironclad" level by default.

- Check every field, parameter, and return type against both docs and implementation
- Flag mismatches with severity (critical/major/minor) and include file:line references
- If a doc says "22 platforms" but the code has 24, that's a finding — count everything
- Verify at implementation level, not just surface level (read the function, not just the docstring)

## Implementation Standards

When implementing a plan or feature, ensure ALL surfaces/components/files mentioned in the plan are addressed comprehensively. Do not scope down or skip items unless explicitly told to. If a plan mentions 12 items, cover all 12.

- Read the entire plan before starting — understand the dependency chain
- Implement in the order specified (or dependency order if unspecified)
- After each logical group of changes, run `source venv/bin/activate && python -m pytest tests/ -x -q` to catch regressions early
- After ALL changes, run `source venv/bin/activate && pre-commit run --all-files` before reporting done
- Never report "done" with failing tests or lint errors

## Safety & Defensive Coding

Never remove, delete, or modify existing working code/config (e.g., triggers, endpoints, registry entries, settings) unless explicitly asked. When fixing bugs, verify your changes don't break adjacent functionality. After making fixes, confirm nothing was inadvertently removed.

- Before editing a file, read it fully and understand what exists
- After editing, mentally diff your changes — did you accidentally delete anything?
- Run `git diff` on your changes and review before committing
- If you're unsure whether something is used, search for references before removing it
- DISABLED_SERVICES in `app/registry/__init__.py` must never be reduced without explicit approval

## Commit & Pre-commit Rules

Pre-commit hooks may take 30-60+ seconds — do not assume timeouts indicate failure. The guardian hook enforces file size limits (650 lines max). Plan for this constraint proactively by splitting large files before committing.

- If guardian blocks a commit for file size, trim docstrings or split the file — don't just suppress the hook
- If ruff-format reformats files you didn't directly change, include them in the commit (that's normal)
- If mypy reports errors, fix them properly with type annotations — don't use `# type: ignore` unless truly necessary
- Pre-push runs `python -m pytest tests/ -x -q` — all tests must pass before push succeeds

## AI Agent Rules (MANDATORY)

**These rules are enforced by pre-commit hooks. Violations will block commits.**

### Version Management

1. **VERSION lives in `app/main.py`** - This is the single source of truth
   ```python
   VERSION = "0.9.0"  # Current version
   ```

2. **When bumping version, you MUST also update CHANGELOG.md**
   - Add entry under `## [x.y.z] - YYYY-MM-DD`
   - Group changes: Added, Fixed, Changed, Removed, Security, BREAKING

3. **Semantic versioning rules:**
   - PATCH (0.9.0 → 0.9.1): Bug fixes only
   - MINOR (0.9.x → 0.10.0): New features, backward compatible
   - MAJOR (0.x.x → 1.0.0): Breaking changes

### Release Process

**Read `docs/operations/RELEASE_GUIDE.md` before any release work.**

1. **Never ship on Friday** - No exceptions
2. **Always run tests before committing**: `make test`
3. **Pre-commit hooks must pass**: `pre-commit run --all-files`

### Code Quality Gates

Pre-commit hooks enforce:
- `ruff` - Linting and formatting
- `mypy --strict` - Type checking
- `release-check` - Version/changelog validation
- `guardian` - Custom quality checks
- File size limits (500KB max)

### Disabled Services (Production Safety)

Services in `app/registry/__init__.py` are managed two ways:

**Always disabled** (trading — HIGH RISK):
- `ibkr` - Interactive Brokers
- `schwab` - Charles Schwab

**Key-gated** (auto-enable when env var is set):
- `blandai` - Voice AI (set `BLANDAI_API_KEY`)
- `twilio` - SMS/Voice (set `TWILIO_ACCOUNT_SID`)
- `hyperbrowser` - Web automation (set `HYPERBROWSER_API_KEY`)

**Do NOT add trading services to key-gated — they stay always-disabled.**
